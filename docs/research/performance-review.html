Executive Summary

The plugin adds ~750–980 ns of overhead per DI invocation (9 heap allocations, ~2 KB). The disabled path is a true zero-cost no-op at ~125 ns (samber/do's own overhead). BuildReport scales near-linearly: ~47 μs for 50 services up to ~590 μs for 500 services. The dominant performance bottleneck is garbage collection — it consumes 43% of CPU cycles in the invocation hot path, driven by per-hook string allocations (serviceKey) and event struct boxing.

The single-RWMutex design is simple and correct but serializes all concurrent writes, adding ~40–70% overhead under parallel invocation load. The concurrency model is safe but not optimized for high-throughput parallel DI containers.

Overall: Production-ready for typical DI workloads GC pressure is the primary scaling risk

♻️Garbage Collection

This is a direct consequence of the 9 allocations per invocation creating short-lived garbage that the collector must constantly mark and sweep.

GC Root Causes

  • String concatenation in serviceKey() — called 2–3× per hook, each producing a new heap-allocated string. This accounts for 6.9% of total allocation space alone.
  • Event struct escaping to heap — Events are appended to a slice and passed to the onEvent callback, preventing stack allocation. Each event carries pointer fields (*float64, *string) that require separate heap boxing.
  • Duration boxing — DurationMs *float64 requires a heap allocation for every timed event. A value type or fixed field would eliminate this.
  • Stack slice growth — r.stack grows dynamically. While typically small (~10-20 entries), the slice backing array is reallocated on growth.
  • Event slice growth — runtime.growslice at 31% shows the events slice exceeds its initial 1024 capacity under benchmark load and triggers expensive full-copy reallocation.

🔀Concurrency Model

Lock Architecture

The recorder uses a single sync.RWMutex to protect all mutable state. This design prioritizes simplicity and correctness over parallelism.

Key Insight: Every hook acquires mu.Lock() (exclusive write lock). Under parallel DI invocation, all goroutines serialize on the mutex. The 56% overhead at 32 threads is expected for a single-writer contention point. If the DI container is invoked from many goroutines concurrently (e.g., HTTP request handlers), this becomes the throughput ceiling.

Concurrency Safety Properties

  • Deadlock-safe: onEvent callback is invoked outside the lock. enrichCapabilities releases the lock before calling do.ExplainInjector to avoid lock-ordering deadlock with samber/do's internal locks.
  • Stack is shared: The invocation stack (r.stack) is shared across all goroutines. Dependency inference from the stack is only correct if invocations are properly nested within a single goroutine. Cross-goroutine invocations may produce incorrect dependency edges (a known limitation).
  • Atomicity: Each hook is atomic — it acquires the lock, performs all mutations, and releases. No torn states are observable.
  • Read concurrency: BuildReport() uses RLock(), allowing concurrent report generation. However, concurrent writes block all reads.

Disk I/O Concerns

  • No bufio.Writer: All file-based exports write directly through os.File without buffering. For NDJSON with thousands of events, this means thousands of individual write(2) syscalls. A bufio.Writer wrapper would batch writes and reduce syscall overhead by 10–100×.
  • Defensive copy before export: Events() copies the entire events slice under RLock before writing. During export, peak memory is 2× the events size (original + copy). For very large event logs, this can cause memory pressure during export.
  • Non-streaming JSON: json.Encoder.Encode(report) marshals the entire report into a single buffer before writing. For reports with thousands of services, this allocates the full JSON payload in memory before any byte hits disk.
  • File creation safety: os.Create truncates existing files. The nolint:gosec annotation acknowledges this is intentional. No atomic rename pattern is used — a crash during write leaves a partial file.
  • Write/close error handling: Correctly prioritizes write errors over close errors — write failure is returned first, close error second. This is good practice.

Client-Side GPU Considerations (HTML Export)

  • The dependency graph is rendered as SVG via DOM manipulation — entirely CPU-rendered by the browser's layout engine. No WebGL, Canvas 2D, or GPU acceleration is used.
  • The Sugiyama layered DAG layout algorithm (rank assignment via Kahn's algorithm, barycenter crossing reduction, median alignment) runs on the main thread in JavaScript. For graphs with 500+ nodes, this may cause a brief UI freeze (~50-200ms).
  • CSS transitions and animations use the browser's compositor (GPU-accelerated for transform and opacity), but this is trivial GPU usage.
  • Optimization opportunity: For very large graphs (1000+ nodes), OffscreenCanvas with WebGL rendering or Web Workers for layout computation would prevent main-thread blocking. Currently not needed for typical DI container sizes.

🔍Key Findings

Critical

Unbounded Event Slice Growth

Critical RAM

The events slice grows without limit for the lifetime of the process. Every invocation, registration, and shutdown appends an Event (~120 bytes each). A long-running service with frequent DI invocations will eventually exhaust memory.

Impact: OOM crash in long-running processes. At 10K invocations/sec, the events slice grows at ~1.2 MB/sec, reaching 4 GB in ~55 minutes.
Recommendation: Add a configurable max-events limit (ring buffer) or periodic truncation API. Alternatively, provide an Events() chan Event streaming model for real-time consumers.

GC Consumes 43% of CPU in Hot Path

Critical CPU GC

The invocation hooks generate 9 short-lived allocations per call, causing GC to consume nearly half of CPU cycles. At high invocation rates, this is the dominant cost of the plugin.

Impact: At 100K invocations/sec, ~200 MB/s of garbage is generated, causing frequent GC pauses and consuming ~43% of CPU.
Recommendation: (1) Replace serviceKey() string concat with a struct map key {scopeID, serviceName} to eliminate 3 string allocations per hook. (2) Use sync.Pool for Event structs. (3) Change DurationMs *float64 to a value type with a sentinel value. (4) Consider a lock-free ring buffer with pre-allocated events.

High

Write-Lock Contention Serializes Concurrent Invocations

High Concurrency

All hooks acquire mu.Lock() (exclusive write lock). Under parallel DI invocation, all goroutines serialize on the mutex, adding ~56% overhead at 32 threads.

Impact: Throughput ceiling for parallel DI containers. At 32 threads, effective parallelism is ~0.64× of theoretical.
Recommendation: (1) Consider per-shard mutexes (shard by scope ID). (2) Use a lock-free MPSC queue for event appending. (3) Separate the events slice (write-hot) from the services map (write-cold) into different locks.

enrichCapabilities Dominates BuildReport

High CPU

do.ExplainInjector accounts for 33.4% of BuildReport CPU time and 42.9% of its allocations. It is called once per scope and internally rebuilds the entire service DAG.

Impact: BuildReport cost is dominated by samber/do's internal introspection, not our own code. For 500 services: ~200μs of the 590μs total.
Recommendation: (1) Cache enrichCapabilities results and invalidate on registration/override. (2) Make capability enrichment opt-in via a config flag. (3) Skip enrichment for scopes with no changes since last BuildReport.

No Buffered I/O for File Exports

High Disk

All file exports write directly to os.File without bufio.Writer. NDJSON export issues one syscall per event.

Impact: For 10K events, ~10K syscalls instead of ~1. Each syscall adds ~1-2μs of context-switch overhead.
Recommendation: Wrap os.File in bufio.NewWriterSize(file, 64*1024) in writeToFile(). Flush before close. Single-line change.

Medium

Cross-Goroutine Dependency Inference May Be Incorrect

Medium Concurrency

The invocation stack (r.stack) is shared across all goroutines. When service A invokes from goroutine 1 while service B invokes from goroutine 2, the stack contains both frames, and the LIFO assumption for dependency inference breaks.

Impact: False dependency edges in the report. Rare in typical DI usage but possible in parallel initialization.
Recommendation: Document this as a known limitation. For correctness, consider a per-goroutine stack using goroutine-local storage (Go doesn't support this natively, so a map[goroutineID][]stackEntry or a context-carried stack would be needed).

Event Slice Growth Triggers Expensive Reallocation

Medium CPU RAM

runtime.growslice accounts for 31% of CPU samples in the hot path. The events slice starts at capacity 1024 and doubles on overflow, requiring a full copy of all existing events.

Impact: Periodic latency spikes when capacity is exceeded. At 1024 events (~120KB), the copy takes ~10μs. At 1M events, ~12ms.
Recommendation: (1) Make initial capacity configurable. (2) Pre-allocate based on expected service count. (3) Use a chunked event list (linked list of fixed-size arrays) to avoid large copies.

Report Query Methods Are O(n) Linear Scans

Medium CPU

ServiceByName(), EventsByService(), EventsByType(), FailedServices(), UnhealthyServices() all perform linear scans. Report.Index() exists for O(1) lookups but is opt-in.

Impact: For reports with 500+ services, each query is ~5μs. Multiple queries compound.
Recommendation: Already addressed via Report.Index(). Consider making Index the default internal representation, or lazy-building it on first query.

Low / Informational

No Atomic File Write (Rename Pattern)

Low Disk

writeToFile uses os.Create which truncates the target file immediately. A crash during write leaves a partial/corrupt file.

Recommendation: Write to a temp file in the same directory, then os.Rename atomically. Low priority — exports are typically one-shot at shutdown.

Render-Blocking Font CSS @import

Low Network

The @import url('https://fonts.googleapis.com/css2...') in the CSS is render-blocking. The browser must fetch this CSS before rendering text content.

Recommendation: Replace with <link rel="preload"> or move to <link> in <head>. Low impact — system fonts are the fallback.

Time.Now() Called Twice Per Hook Pair

Low CPU

Each before/after hook calls time.Now() independently (~25-40ns each). Duration is computed from the difference.

Recommendation: Acceptable. time.Now() uses vDSO and is fast. The monotonic clock is already used internally by Go's time.Sub().

Diagram Export Uses fmt.Fprintln Per Line

Low Disk

Mermaid and PlantUML exports call fmt.Fprintln for each line, resulting in one write per line.

Recommendation: Use a strings.Builder to batch all lines, then single write. Minor improvement.
