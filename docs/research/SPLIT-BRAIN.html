What is a Split-Brain?

A split-brain occurs when the same piece of information exists in two or more places in the codebase with no mechanism to keep them in sync. When one copy is updated but the other isn't, consumers reading from the stale copy get wrong answers. This is the most dangerous class of data-model bug because it's invisible to the compiler — the code compiles, tests pass, but the data lies.

The most critical finding (#1) is proven with a runnable test: a report where Status="active" but InvocationError="boom" passes Validate(), is missed by FailedServices(), and is silently preserved by MigrateReport().

01 Status Field vs. Derived Status HIGH

The service lifecycle status is stored as a denormalized field AND independently computable from error pointers — but nothing enforces their consistency.

SB-01 ServiceInfo.Status is a stored field that can contradict its own error pointers

▶
The Problem

ServiceInfo.Status is a stored ServiceStatus enum field. It's supposed to always equal the output of ServiceInfo.DeriveStatus(), which computes the status from InvocationError, ShutdownError, ShutdownAt, and FirstInvokedAt.

But nothing enforces this. The Status field can be set to any value via JSON deserialization or stale migration input, and the report still passes Validate(). This creates two sources of truth for "what lifecycle state is this service in?":

SOURCE A (stored): ServiceInfo.Status ──read by──▶ FailedServices(), Diff.StatusChanged, HTML template SOURCE B (derived): ServiceInfo.DeriveStatus() ──read by──▶ MigrateReport() (only when Status is empty!) GUARD: Validate() ──────────────────▶ checks COUNTS only, never checks Status == DeriveStatus()
Why this matters: FailedServices() checks Status.IsError(). If Status is stale, a broken service is invisible to monitoring dashboards, alerting, and regression tests.
Evidence — Exact Code Locations

02 Dual Error Detection HIGH

Two independent functions answer the same question — "does this service have an error?" — from different data, and they can disagree.

SB-02 diff.go:hasError() checks raw pointers; Status.IsError() checks the enum

▶
The Problem

There are three ways to check if a service is in an error state, each reading different data:

Function File:Line Checks Used by
ServiceStatus.IsError() types.go:80 Status enum value FailedServices()
hasError(svc) diff.go:100 InvocationError != nil || ShutdownError != nil Diff() → HasNewError
ServiceInfo.HasHealthError() service.go:36 HealthCheckError != nil (available but unused in core)

The first two represent the same concept ("does this service have a lifecycle error?") but read from different data. When Status is stale (see SB-01), they disagree:

// diff.go:83-98 — compareService uses hasError() func compareService(prev, other ServiceInfo) (ServiceDiff, bool) { diff := ServiceDiff{ HasNewError: !hasError(prev) && hasError(other), // ← raw pointer check } ... } // diff.go:100-102 — hasError bypasses Status entirely func hasError(svc ServiceInfo) bool { return svc.InvocationError != nil || svc.ShutdownError != nil // ← ignores Status field } // report.go:201 — FailedServices uses Status.IsError() if s.Status.IsError() { // ← enum check, trusts stored Status failed = append(failed, s) }
Concrete scenario: A service has InvocationError="boom" but stale Status="active". Diff.HasNewError says true (it checks the pointer). FailedServices() says false (it checks the enum). They contradict each other on the same report.
How to Fix
  1. Delete hasError() in diff.go:100-102.
  2. Replace its call site in diff.go:89 with:
    // Uses the Status field consistently with FailedServices() HasNewError: !prev.Status.IsError() && other.Status.IsError(),
  3. Or, for maximum safety, use DeriveStatus() instead of the stored field: !prev.DeriveStatus().IsError() && other.DeriveStatus().IsError(). This makes the check immune to stale Status even before SB-01 is fixed.

03 serviceRecord ↔ ServiceInfo Parallel Model MEDIUM

16 fields are manually mirrored between an internal mutable struct and a public immutable snapshot with no compile-time guarantee that all fields are mapped.

SB-03 Manual field-by-field copy between serviceRecord and ServiceInfo

▶
The Problem

The codebase has two structs representing the same concept — a service's observed state:

Struct File Role Mutable? Field count
serviceRecord recorder.go:40 Internal working state (live updates from hooks) Yes (behind mutex) 16
ServiceInfo service.go:6 Public immutable snapshot (in Report) No (value type) 20

The bridge is a 26-line manual copy in buildServicesLocked(). Every field on serviceRecord must be explicitly listed:

// report_builder.go:50-74 — manual field-by-field copy services = append(services, ServiceInfo{ ServiceRef: ServiceRef{ ServiceName: rec.serviceName, ScopeID: rec.scopeID, ScopeName: rec.scopeName, }, Status: computeServiceStatus(rec), ServiceType: rec.serviceType, RegisteredAt: rec.registeredAt, FirstInvokedAt: rec.firstInvokedAt, InvocationCount: rec.invocationCount, InvocationOrder: rec.invocationOrder, FirstBuildDurationMs: rec.firstBuildDurationMs, Dependencies: deps, Dependents: svcDependents, ShutdownAt: rec.shutdownAt, ShutdownDurationMs: rec.shutdownDurationMs, ShutdownError: rec.shutdownError, InvocationError: rec.invocationError, IsHealthchecker: false, // ← not on serviceRecord at all IsShutdowner: false, // ← not on serviceRecord at all LastHealthCheckAt: rec.lastHealthCheckAt, HealthCheckError: rec.healthCheckError, HealthCheckCount: rec.healthCheckCount, })
The risk: If a developer adds lastInvokedAt *time.Time to serviceRecord (recorder.go) but forgets to add the corresponding field to ServiceInfo and the copy in buildServicesLocked, the data is silently lost in reports. The exhaustruct linter catches missing fields in struct literals, but does NOT verify the copy is semantically complete.
How to Fix
  1. Extract a conversion function to centralize the mapping in one place:
    // Put this in hooks.go next to newServiceRecordCore func serviceRecordToInfo(rec *serviceRecord) ServiceInfo { return ServiceInfo{ ServiceRef: ServiceRef{ ServiceName: rec.serviceName, ScopeID: rec.scopeID, ScopeName: rec.scopeName, }, ServiceType: rec.serviceType, RegisteredAt: rec.registeredAt, FirstInvokedAt: rec.firstInvokedAt, InvocationCount: rec.invocationCount, InvocationOrder: rec.invocationOrder, FirstBuildDurationMs: rec.firstBuildDurationMs, ShutdownAt: rec.shutdownAt, ShutdownDurationMs: rec.shutdownDurationMs, ShutdownError: rec.shutdownError, InvocationError: rec.invocationError, LastHealthCheckAt: rec.lastHealthCheckAt, HealthCheckError: rec.healthCheckError, HealthCheckCount: rec.healthCheckCount, } }
    Then call it in buildServicesLocked and fill in the derived fields (Status, Dependencies, Dependents, IsHealthchecker, IsShutdowner) after.
  2. Add a test that populates every serviceRecord field, calls BuildReport(), and asserts every field appears in the corresponding ServiceInfo. This catches forgotten fields at test time.

04 Provider Icon Metadata Asymmetry LOW

ProviderType has an Icon() method; ServiceStatus and EventType do not. Icons for the latter are hardcoded in a separate file with no compiler enforcement.

SB-04 Inconsistent display-metadata derivation across enum types

▶
The Problem
Enum Type Has Icon() method? Icon source
ProviderType ✓ Yes types.go:53 — method on enum
ServiceStatus ✗ No metadata.go:42 — hardcoded map
EventType ✗ No metadata.go:48 — hardcoded map

BuildTypeMetadata() derives provider icons by calling ProviderType.Icon() — a method on the enum type. But status and event icons/labels are hardcoded string literals in the same function. Adding a new ServiceStatus value in types.go without updating metadata.go produces a silently missing icon in the HTML visualization.

How to Fix
  1. Add func (s ServiceStatus) Icon() string to types.go following the ProviderType.Icon() pattern.
  2. Add func (e EventType) Label() string and func (e EventType) Color() string similarly.
  3. Update BuildTypeMetadata() to call these methods instead of hardcoding strings.
  4. This makes the metadata struct the single consumer of the methods, not a parallel source of truth.

05 Triplicated JSON Encoding Paths LOW

Three independent code blocks encode reports as JSON with identical logic but no shared helper.

SB-05 Report.WriteJSON(), Plugin.WriteReportJSON(), and ExportFilteredToFile each encode independently

▶
The Problem
// report.go:289-299 — Path 1 func (r Report) WriteJSON(writer io.Writer) error { enc := json.NewEncoder(writer) enc.SetIndent("", " ") return enc.Encode(r) } // plugin.go:142-153 — Path 2 (duplicates Path 1) func (p *Plugin) WriteReportJSON(writer io.Writer) error { report := p.Report() enc := json.NewEncoder(writer) enc.SetIndent("", " ") return enc.Encode(report) } // plugin.go:174-184 — Path 3 (duplicates again for filtered) return writeToFile(path, func(w io.Writer) error { enc := json.NewEncoder(writer) enc.SetIndent("", " ") return enc.Encode(filtered) })

If indentation or error-wrapping changes in one path, the others silently diverge. Plugin.WriteReportJSON() could simply delegate to Report.WriteJSON().

How to Fix
  1. Make Plugin.WriteReportJSON() call p.Report().WriteJSON(writer).
  2. Make ExportFilteredToFile call filtered.WriteJSON(writer) inside the writeToFile closure.
  3. One encoding path, one source of truth for indentation and error wrapping.

The scope's human-readable name is copied into every struct that references a scope, rather than looked up from the single scope registry.

  3. This centralizes scope-name resolution to scopeMeta as the single source.

Glossary for Junior Developers

Key terms used in this audit. Read these before attempting any fix.

Denormalized field
A field whose value is computed from other fields and then stored, rather than computed on-demand. Example: Report.EventCount is denormalized from len(Report.Events). Safe when guarded by Validate() or finalizeDenormalized(). Dangerous when not.
Split-brain
When the same information has two representations that can disagree. The classic pattern: a stored field + a derivation function, with no validation that they match.
Single source of truth
A design where each fact is stored/computed in exactly one place. All consumers read from that one place. This audit's goal is to move toward this for every finding.
Drift
When two copies of the same data get out of sync. Usually caused by updating one copy (e.g., the error pointer) but not the other (e.g., the Status field).
serviceRecord (internal)
The mutable working state for a service, updated live by hooks. Lives in Recorder.services map behind a mutex. Never serialized directly.
ServiceInfo (public)
The immutable snapshot of a service in a Report. Built from serviceRecord at report time. Serialized to JSON.
DeriveStatus()
The function that computes lifecycle status from error pointers and timestamps. Priority: invocation_error > shutdown_error > shutdown > active > registered. Defined in report_helpers.go:72.

How This Audit Was Conducted

Every .go file in the package root was read in full. The following patterns were systematically searched for:

  • Duplicate type definitions — structs representing the same concept in multiple places
  • Denormalized fields — stored values that duplicate computable data (counts, statuses, flags)
  • Multiple derivation paths — the same computation in 2+ functions
  • Validation gaps — Validate() not covering all invariants
  • Enum metadata asymmetry — some enums have methods, others use parallel maps
  • Manual field copying — struct-to-struct conversion without compile-time enforcement

The critical finding (SB-01) was verified with a runnable proof-of-concept test that demonstrates Validate() passing on a self-contradictory report.

Areas checked and found clean: Event/Phase enums (single source in types.go), buildReportFromCore invariant (all construction paths route through it), finalizeDenormalized (recomputes all 8 aggregate fields from core data), ServiceRef identity (embedded, not duplicated), scopeKey removal (all callers use serviceKey).
