Arguments FOR

4 arguments
01 Same author, aligned philosophy, no governance friction MODERATE ▼
Both projects are by larsartmann, use Go 1.26.3, share the same build toolchain, and depend on samber/do v2. Version bumps, Go upgrades, and samber/do API changes can be coordinated without cross-org coordination. If cmdguard breaks something, you fix both yourself.
02 cmdguard's DI Scope wraps the same Injector auditlog hooks into MODERATE ▼
cmdguard's Scope wraps do.Injector and provides Provide, Invoke, Shutdown, HealthCheck. If a consumer uses cmdguard to build their CLI, auditlog could theoretically detect that scope and attach automatically. However, this is a consumer-level integration, not a library-level dependency. auditlog doesn't need to import cmdguard to do this.
03 Rich output formats (12+) could enhance export options MODERATE ▼
cmdguard integrates go-output for 12+ formats (JSON, CSV, YAML, D2, Mermaid, HTML, etc.). If auditlog ever wanted to export the services table as CSV or the dependency graph as D2/Mermaid text, cmdguard re-exports that capability.

However, this was already analyzed and rejected in the go-output-adoption.md brainstorming document — TableData is [][]string, a fundamental mismatch for auditlog's rich domain structs. Importing cmdguard for this would add even more weight than go-output alone.
04 OpenTelemetry middleware for command-level tracing MODERATE ▼
cmdguard has TelemetryMiddleware that creates OTel spans per command execution. If auditlog wanted to correlate DI lifecycle events with CLI command traces, both using OTel would be elegant.

But auditlog is a library plugin, not a CLI application. It doesn't execute commands. The tracing correlation would happen at the consumer level — the consumer's CLI tool would use cmdguard's telemetry AND auditlog's events. Neither needs to import the other.

Arguments AGAINST

7 arguments
01 Zero API overlap — cmdguard solves CLI problems, auditlog is a DI plugin CRITICAL ▼
cmdguard provides: NewCLI[T], NewCommand[T,F], struct-tag flag parsing, Cobra wiring, help rendering, shell completion, man pages, middleware chains, interactive prompts, styled error output.

auditlog needs: do.InjectorOpts hooks, timestamped event recording, dependency graph inference, JSON/HTML export.

These are completely disjoint problem domains. There is no function, type, or pattern in cmdguard that auditlog could call or reuse. Importing it would be like importing a web framework into a math library.
02 Dependency explosion: 2 deps → 15+ transitive deps CRITICAL ▼
auditlog currently depends on samber/do v2 and a-h/templ — 2 direct deps, ~3 transitive. That's it.

cmdguard pulls in: spf13/cobra, spf13/pflag, charm.land/fang, charm.land/huh, charm.lipgloss, charm.bubbletea, charm.bubbles, charm.glamour, go-output (+ 7 sub-modules), otel/trace, go-toml, yaml, and ~25 more transitive deps.

That's a +650% increase in dependency surface for zero functional gain. Every one of those deps is a supply-chain vector, a potential version conflict, and a compile-time cost — for nothing.
03 Violates auditlog's design principle: minimal, focused dependencies STRONG ▼
The .golangci.yml has depguard configured to only allow $gostd, samber, templ, and samber-do-auditlog. This is intentional — auditlog is a focused plugin that should be cheap to adopt. Adding cmdguard would force opening depguard to allow larsartmann/cmdguard + all its transitive deps, weakening the supply-chain guardrail for zero benefit.
04 Cobra is CLI infrastructure — inappropriate for a DI plugin library STRONG ▼
cmdguard's core dependency is spf13/cobra. Importing cmdguard transitively imports cobra into every binary that uses auditlog — even if that binary has no CLI at all (e.g., a long-running server using samber/do for DI).

A DI audit plugin has no business pulling in a CLI command framework. Consumers who build CLIs can import cmdguard themselves. The dependency should flow: consumer → cmdguard and consumer → auditlog, not auditlog → cmdguard.
05 The "Scope" integration is consumer-level, not library-level STRONG ▼
Both projects use samber/do v2. cmdguard's Scope wraps do.Injector. auditlog hooks into do.InjectorOpts. The natural integration point is:

consumer creates cmdguard CLI → gets Scope → passes Scope.Injector() to auditlog.New()

This works today without either library importing the other. auditlog accepts any do.Injector. cmdguard exposes Scope.Injector(). The integration is 3 lines of consumer code.
06 No code reuse, no type reuse, no pattern reuse MODERATE ▼
Scanning cmdguard's entire v2 package: no struct, interface, function, or type that auditlog could reuse. The generic CLI[T], Command[T,F], Scope, FlagRegistry, BranchingFlowContext, middleware chain — none apply to auditlog's domain (event recording, service aggregation, dependency inference, report generation).

The only shared type is do.Injector — which comes from samber/do, not cmdguard.
07 Compiles and ships cobra/pflag to non-CLI consumers unnecessarily MODERATE ▼
auditlog is used by any Go application that uses samber/do for DI — servers, workers, batch processors, not just CLIs. Importing cmdguard would force every such consumer to compile cobra, pflag, fang, huh, bubbletea, glamour, and the entire TUI stack even though they have no terminal interaction. This is a significant compile-time and binary-size tax for zero utility.

Analysis

Key factors

✕ Domain mismatch
cmdguard is a CLI framework. auditlog is a DI observability plugin. They share samber/do but operate at completely different abstraction levels. No shared domain, no shared types, no shared problem.
✕ Dependency cost
2 deps → 15+ deps. Cobra, fang, huh, bubbletea, glamour, otel, go-output — an entire TUI+CLI+observability stack dragged into a DI plugin that exports JSON and HTML.
● Consumer-level integration
The natural bridge is consumer code: auditlog.New() + scope.Injector() from cmdguard. Works today, no imports needed. This is the correct layering.
● Already rejected go-output
The go-output-adoption.md analysis rejected adding go-output for the same reasons. cmdguard re-exports go-output, making it an even heavier version of an already-rejected dependency.
Verdict — Do Not Adopt

cmdguard and auditlog are complementary tools
for the same ecosystem, not dependencies of each other.

The correct relationship is horizontal, not vertical. A consumer builds a CLI with cmdguard, wires samber/do for DI, and plugs in auditlog to observe the container. Both libraries sit on samber/do v2 as their shared foundation. Neither should import the other.

Importing a CLI framework into a DI plugin is an architectural category error. It would explode the dependency surface by 650% for zero lines of code reuse, violate auditlog's depguard policy, and force every consumer — even non-CLI servers — to compile cobra, pflag, and an entire TUI stack they don't need.

The right place for integration is consumer code. Three lines: create the cmdguard scope, extract the injector, pass it to auditlog. Done.

Revisit

When to revisit this decision

This decision is stable. Revisit only if one or more of these conditions materialize:
  • cmdguard extracts its Scope DI layer into a standalone package without cobra/fang/TUI deps, and auditlog needs it
  • auditlog adds CLI subcommands (e.g., auditlog report, auditlog serve) and becomes a CLI tool itself — then cmdguard would be a consumer dependency, not a library dependency
  • cmdguard adds an auditlog integration module (e.g., cmdguard/auditlog) that auto-attaches auditlog to scopes — this would live in cmdguard's repo, not auditlog's
  • A shared samber/do-ecosystem utilities package emerges that both could depend on without pulling in CLI or TUI deps
