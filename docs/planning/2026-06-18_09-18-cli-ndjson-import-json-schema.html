The keystone blocker Reconstructing a Report (with Services, ScopeTree, and all denormalized counts) from a bare []Event is the single piece of work that unlocks NDJSON import, the CLI, and schema-validated diffing. The logic currently lives stateful inside Recorder hooks — it must be extracted into a pure, testable ReplayEvents function. Without this, NDJSON import yields only an event bag, useless for diffing or HTML re-rendering.

01The three features — why they belong together

1. NDJSON Import

Load events back from a saved .ndjson file. Currently the library is write-only: writeEventsNDJSON in export.go:12 emits events but nothing reads them. Import + replay closes the loop and makes Report.Diff usable across time.

2. JSON Schema file

schema/report.schema.json describes the v0.2.0 wire format machine-readably. Report.Validate() in report.go:46 only checks internal consistency; it cannot catch type/shape violations in a hand-edited or third-party JSON. The schema is the contract.

3. Stand-alone CLI

A cmd/auditlog binary with import, export, validate, diff, info subcommands. Today the repo is library-only (flake.nix:44-59 explicitly ships a README stub, not a binary). CLI turns every library API into an end-user command.

The synergy The CLI is a thin shell over library APIs — but it has nothing to call without NDJSON import (you can't load saved data) and no contract without JSON Schema (you can't validate what you load). Building any one alone is half a feature. Building all three in the order below gives a fully round-trippable telemetry tool.

♦What the codebase already gives us for free

+
All export formats already exist WriteJSON, WriteNDJSON, WriteHTML, WriteMermaid, WritePlantUML — the CLI just wires inputs to these.
+
Report.Diff already exists Directional diff with added/removed/changed services and event-count delta. Extensions (deps, scopes) are additive.
+
MigrateReport handles old JSON Version-agnostic re-derivation means LoadReportFromJSON gets v0.1.0→v0.2.0 upgrade for free.
+
Event struct is the wire unit NDJSON is one Event per line — decoding is a one-liner per line. The hard part is reconstruction, not parsing.
+
buildReportFromCore is the single finalizer Any new construction path (including replay) just builds a core state and hands it to buildReportFromCore — counts cannot drift.
+
samber/ro is depguard-allowed .golangci.yml permits github.com/samber/*, so a reactive-streams adapter (EventsAsObservable) imports cleanly with zero policy churn. Fills the live-streaming gap with first-class Rx operators (filter, debounce, window, zip) instead of bespoke chan/iter.Seq plumbing.

02Pareto breakdown — what delivers the most for the least

Following the 80/20 rule applied recursively:

Tier Yield What's in it Why it's there
T1 KEYSTONE 1% → 51% ReplayEvents([]Event) (Report, error) — pure function that reconstructs Services + ScopeTree from a flat event stream by re-using the Recorder's state-machine logic, plus ReadEvents(io.Reader) ([]Event, error). Without this, NDJSON import returns a useless event bag (no Services to diff, no scope tree to render). Every other feature depends on being able to load a saved Report. This is the only piece with non-trivial logic.
T2 OPERATIONAL 4% → 64% LoadReport(path) with format auto-detect, cmd/auditlog skeleton with cobra, import & export subcommands, JSON Schema v0.2.0 file + embedded validator. Converts the keystone into a usable product: a user can now auditlog import events.ndjson -o report.html and get a visualisation. Schema gives a machine-checked contract.
T3 CUSTOMER VALUE 20% → 80% validate, diff, info subcommands; Report.Diff extensions (dependency edges, scope tree); golden/fuzz tests; nix build binary. This is where the diff-across-time promise is actually delivered, and where the library gains a buildable artifact. Tests here de-risk the keystone.
T4 POLISH 80% → 100% Cross-compile CI matrix, README/AGENTS/FEATURES docs, cli-workflow.md example, CHANGELOG entry. Discovery, distribution, and maintainability. Without these nobody knows the features exist and future contributors hit undocumented traps.

03Honesty constraints & risks

Risk A — Capability flags cannot be reconstructed

IsHealthchecker and IsShutdowner on ServiceInfo are populated by enrichCapabilities() in BuildReport(), which calls do.ExplainInjector(scope) on a live *do.Scope reference. A replayed Report has no live container. These two bools will always be false on imported Reports. Mitigation: document loudly in AGENTS.md and add a Reconstructed bool flag on Report so consumers can detect this state.

Risk B — diff.go doc currently lies

The doc comment at diff.go:43 claims dependency edges are compared, but compareService in diff.go:83-98 only looks at Status, InvocationCount, HealthCheckCount, and an error-state boolean. Either fix the doc or implement the comparison. The plan does both: extend the diff and correct the doc.

Risk C — NDJSON has no schema version

Each Event line is bare JSON; there is no header and no version marker. An importer cannot detect "this NDJSON was emitted by a future v0.3.0 with new fields". Future Event fields with omitempty will silently round-trip; field removals will fail loudly. Mitigation: add a separate .auditlogmeta sidecar convention (out of scope here; flagged for roadmap).

Risk D — Recorder refactor risk

Extracting ReplayEvents means either (a) refactoring the Recorder to share logic, or (b) duplicating the state machine. Option (a) touches the hottest code path (every hook). Mitigation: prefer (b) for v1 — duplicate with a golden-test gate, refactor later when both paths are proven equivalent. The library is ~2,500 LOC; a few hundred lines of parallel logic is acceptable.

07Recommended execution order (sequential critical path)

The plan can be parallelised within tiers, but the critical path below gives the fastest time-to-first-user-value if executed strictly serially.

1

Foundation — ~5h

T1 (extract replay logic) → T2 (ReplayEvents) → T3 (NDJSON reader) → T15 (replay golden tests). After this step you can load any saved NDJSON file back into a Report in Go code. No CLI yet, but the keystone is proven.

2

Operational layer — ~6h

T4 (Loader API) + T5 (CLI skeleton) → T6 (import) + T7 (export) + T8 (JSON Schema draft) → T9 (schema validation). After this step auditlog import events.ndjson -o report.html works end-to-end.

3

Customer value — ~8h

T10 (diff deps) + T11 (diff scopes) → T12 (validate) + T13 (diff) + T14 (info) + T16 (NDJSON tests) + T17 (CLI tests) + T18 (nix binary) + T26 (samber/ro adapter). After this step the diff-across-time promise is delivered, the binary is installable, and live event streams compose into Rx pipelines.

4

Polish & distribution — ~5h

T19 (cross-compile CI) + T20 (smoke) + T21 (README) + T22 (AGENTS.md) + T23 (FEATURES/TODO) + T24 (workflow doc) + T25 (CHANGELOG). Ship.

Parallelisation opportunity T1+T2 must be serial. But T3 (NDJSON reader), T5 (CLI skeleton), T8 (JSON Schema draft), and T26 (samber/ro adapter) can all proceed in parallel with T1/T2 since none depend on the replay engine. That drops the critical-path wall-clock time from ~25h to ~18h on a 2-engineer team.

08Explicitly out of scope (flagged for roadmap)

  • Streaming event consumer for saved files — iter.Seq[Event] over NDJSON, or channels/SSE/WebSocket endpoints. Note: the live streaming gap is now closed by T26 (samber/ro adapter); only file-backed streaming remains out of scope.
  • NDJSON sidecar metadata (schema version embedded in stream). Mitigation for Risk C above; deferred until a real v0.3.0 schema change forces it.
  • HTML diff visualisation (rendering a DiffResult as a side-by-side HTML page). Powerful but a separate feature; the CLI emits text + JSON for now.
  • Remote report sources (HTTP/S3 loading). The CLI takes file paths; remote fetching belongs in a wrapper or future --source url flag.
