Architecture Deepening Opportunities

samber-do-auditlog — 2026-06-10

Summary

3 candidates found. The codebase is already well-structured for its size (1 package, 1757 LOC). The opportunities below are forward-looking — they prepare the library for growth rather than fix current problems.

1. Report Filtering via Functional Options

recorder.go, types.go

Strong

Before

Report() → returns ALL data
No way to filter
Consumer must filter client-side

After

Report(opts...ReportOption)
WithServiceFilter("db")
WithTimeRange(from, to)
WithEventTypes(Invocation)

Problem: As the library gets used in larger applications with hundreds of services, Report() will produce increasingly large payloads. Consumers who only need a subset must deserialize the full report and filter client-side.

Solution: Add ReportOption functional options to Report() and BuildReport(). The options pre-filter at the recorder level, avoiding unnecessary serialization.

Benefits: Leverage — one clean interface hides complex filtering. Locality — filtering logic lives in one place, not scattered across consumers.

2. Real-time Event Streaming Callback

plugin.go, recorder.go

Worth Exploring

Before

Capture-only model
Export = batch operation
No real-time observability

After

Config.EventHandler func(Event)
Each event streams to handler
Enables live dashboards, logging

Problem: The library captures events but only exposes them through batch export. Users who want real-time observability (e.g., logging each service invocation as it happens) must poll Report().

Solution: Add an optional EventHandler func(Event) to Config. When set, each captured event is also forwarded to the handler. This is a thin seam — zero cost when not configured.

Benefits: Enables integration with logging/metrics systems without changing the export model. The Recorder deepens — more behavior behind the same interface.

3. Versioned Report Schema with Migration

types.go, recorder.go

Speculative

Before

SchemaVersion = "0.1.0"
No migration path
Consumers break on change

After

Versioned schema types
Migrate(oldVersion, data) → Report
Backward-compatible consumers

Problem: SchemaVersion exists but is only a string constant. If the report schema changes in v0.2.0, old exports become unparseable. This is acceptable for ALPHA but becomes a liability at v1.0.

Solution: Define a versioned schema contract. Add a MigrateReport(version, data) (Report, error) function that can upgrade old exports to the current schema.

Benefits: Long-term stability. Consumers can rely on the schema contract surviving version bumps. Postpones until v1.0 planning.

Top Recommendation

Candidate #1 (Report Filtering) — tackle first. It's the highest leverage: a clean interface change that scales the library for real-world use without breaking existing consumers. The functional options pattern is idiomatic Go and well-understood by the ecosystem.

Candidate #2 (Event Streaming) is a close second — it complements #1 by adding real-time capability. Together they transform the library from "batch export tool" to "comprehensive DI observability layer".

Candidate #3 (Versioned Schema) should wait until v1.0 planning begins.
