02Partially Done

Lint

16 lint warnings in live/ package

live/server.go, live/hub.go, live/server_test.go

The code compiles and tests pass, but golangci-lint reports warnings: exhaustruct (struct literal completeness), varnamelen (w/r parameter names — standard net/http convention), errchkjson (unchecked Encode error on healthResponse with float64), gci (import ordering), modernize (interface{} → any). The w/r parameter names are idiomatic Go for http.Handler and match the existing project patterns.

Dashboard

JS is a first pass — not as polished as the static HTML

live/dashboard.js

The live dashboard JS does incremental rendering (services table, events table, waveform, stats, legend, DAG) but lacks some features from the static templ version: collapsible scope tree, service dependency detail view, scope-scoped service grouping, the "Show all" pagination, the lifecycle waveform with per-event duration bars, and the Sugiyama layered graph. The static dashboard is the gold standard — the live version is functional but simpler.

CSS

base_css.go is a manual extraction, not derived from the templ

live/base_css.go

The CSS was hand-extracted from the static html.templ rather than shared. If the static dashboard CSS changes, the live version will drift. Consider extracting the shared CSS into a common package or using go:embed on a shared CSS file.

03Problems (What I Forgot / Could Have Done Better)

Forgotten

No demo/main.go example

live/demo/

The go-workflow-auditlog/live/ has a demo/ package that runs a multi-step pipeline with intentional delays so you can watch the dashboard update in real time. I didn't create an equivalent demo for samber-do-auditlog. This is the most important UX artifact — without it, nobody can see the dashboard in action without writing their own setup.

Forgotten

No integration with the example/ app

example/main.go

The example/ app already runs an HTTP server and exports HTML to files. It would be natural to add a --live flag or integrate the live dashboard into the existing example, showing the dashboard alongside the DI container lifecycle.

Omission

AGENTS.md not updated with live/ architecture

AGENTS.md

The AGENTS.md Architecture section lists all source files but doesn't mention the live/ sub-package. The Data Flow, Gotchas, and Testing Patterns sections should be updated to reflect the new SSE streaming capability.

Omission

FEATURES.md and TODO_LIST.md not updated

FEATURES.md, TODO_LIST.md

The live dashboard is a major new feature that should be listed in FEATURES.md. TODO_LIST.md should track the known improvements (demo app, scope tree, pagination, CSS sharing).

Missed

CORS and CSP headers for the live dashboard

live/server.go

The static HTML dashboard has a strict CSP meta tag. The live dashboard's HTML has the CSP meta tag in the template, but the SSE endpoint doesn't set CORS headers. If someone embeds the dashboard in an iframe or accesses it from a different origin, it will break. The CSP is embedded in the HTML via meta tag, which is correct, but should be verified.

Polish

Report endpoint uses encoding/json, not encoding/json/v2

live/server.go:handleReport, handleHealth

The go-workflow-auditlog/live/ uses encoding/json/v2 + encoding/json/jsontext for serialization. samber-do-auditlog has an explicit policy against json/v2 (requires GOEXPERIMENT=jsonv2). The current implementation uses stdlib encoding/json, which is correct for the project constraints.

06Open Questions

Q1: Should live/ be a separate Go module (like go-workflow-auditlog/live/) or stay in the same module? go-workflow-auditlog/live/ is a separate go.mod with replace directives for the parent. samber-do-auditlog explicitly does NOT modularize (per AGENTS.md: "Do NOT modularize — Project is 1 package, ~2,500 LOC"). The live/ sub-package currently lives in the same module, which means it can access unexported types. If it were a separate module, it could only use exported APIs. Which approach do you prefer?
Q2: Should the live dashboard reuse the existing templ-based HTML or build its own? The static dashboard (html.templ) is a 1000-line templ template with all CSS/JS inline. The live dashboard has its own separate HTML template, CSS, and JS. Should we refactor to share the templ, or keep them separate? Sharing would reduce drift but增加 complexity. The go-workflow-auditlog approach keeps them separate.
Q3: Should the demo use the existing example/ ride-sharing domain or create a simpler one? The example/ has 19 features with a ride-sharing domain (drivers, passengers, matching). A simpler demo (3-4 services: db, cache, api, notifier) might be clearer for showing the live dashboard. Or should we extend the existing example to also serve the live dashboard?
