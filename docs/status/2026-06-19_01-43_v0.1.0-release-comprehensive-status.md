## D. TOTALLY FUCKED UP

1. **`invocationOrder` bug shipped for 4+ commits** — The most critical finding. Every replayed report had all services at invocation order 0. CSV export, HTML timeline, and any consumer relying on build ordering got silently wrong data. The bug existed from the original replay engine implementation through 4+ commits before being caught by the brutal self-review.

2. **Golden file committed stale** — The golden file test is supposed to catch output changes. It was never regenerated after the `invocationOrder` bug fix. CI was green while the golden file itself was wrong. The test validated against incorrect expected output. This happened TWICE during the session — the golden file kept reverting in the working tree because `go generate` and `golangci-lint --fix` fought over `html_templ.go` formatting.

3. **Three ghost API functions shipped as public surface** — `LoadReportFromJSON`, `LoadReportFromNDJSON`, and the CLI's `loadFromReader` (which reimplemented `LoadReportFromReader`) — all exported, documented, tested, with zero non-test consumers. They added maintenance burden and confused the API surface.

4. **Two different ServiceRef sort orderings** — `compareByName` sorted ScopeName primary; `sortServiceRefs` sorted ServiceName primary. Dependencies in reports were sorted differently from diff results for the same data. A split brain that would have been invisible in testing because each path's tests only checked its own ordering.

5. **`golangci-lint --fix` corrupts generated files** — The `gci` formatter merges `html_templ.go` imports into a grouped block, but `templ generate` produces separate import lines. The `_templ\.go$` exclusion exists in `.golangci.yml` but `--fix` ignores it (golangci-lint v2 bug). This caused repeated churn where `--fix` would "fix" generated files, then `go generate` would revert them, creating false diff in the stale-generation CI check.

## E. WHAT WE SHOULD IMPROVE

### Architecture

- **Single source of truth maintained** — `buildServicesFromMap` is the single service-assembly function. `popStackFrame` is the single stack-pop. `CompareServiceRefs` is the single sort. `scopeMeta` is the single scope type. This is good architecture. Maintain these invariants aggressively.
- **Strict enum unmarshaling at the type boundary** — Currently only `ReadEvents` validates enums. `json.Unmarshal` into a `Report` (via `MigrateReport`) silently accepts garbage. Add `UnmarshalJSON` to all four enum types.
- **`Report.Version` should be typed** — Currently a bare `string`. A `SchemaVersion` type with a constructor and validation would make invalid versions unrepresentable.

### Testing

- **Property-based replay round-trip** — The replay engine should be a proper inverse of the recording path. This is the single most valuable test we could add.
- **Schema drift detection is in place** — `TestSchemaNoDrift` catches type/schema drift. Good.
- **Coverage is 95.2%** — Above the 95% gate but not by much. The new `MergeReports` and `stats` subcommand have basic coverage but could use edge-case tests.

### Process

- **Never use `golangci-lint --fix` globally** — It reformats generated files. Use `--fix` only on specific non-generated files, or use `gofmt -w` directly.
- **Regenerate golden files after ANY rendering pipeline change** — The golden file reverted twice during this session because of formatting conflicts.
- **Update TODO_LIST.md and FEATURES.md after every release** — Both are stale relative to v0.1.0.

## G. Top Question

**"Should the replay engine and the live recording path share a common state holder?"**

They already share: `buildServicesFromMap`, `popStackFrame`, `recordDependencyFromStack`, `newEventFromRef`, `newServiceRecordCore`, `serviceRecordToInfo`, `sortedScopes`, `scopeMeta`, and `buildServiceDeps`. The remaining divergence is in the state holders (`Recorder` vs `replayState`) and the locking model (`sync.RWMutex` vs lock-free).

The question is: should we go further? Should `replayState` embed `Recorder`? Or should we extract a `coreRecorder` that both wrap?

**Answer: No.** The current design is the correct level of sharing. The shared functions are pure and stateless, while the stateful logic is path-specific. Pushing further unification would couple the replay path to mutex semantics it doesn't need. The `replayState` is intentionally lightweight (no mutex, no atomic counters except `invocationSeq`, no `*do.Scope` references). Making it embed `Recorder` would add 4 fields it doesn't use and a locking model it doesn't need.

The real risk is not under-sharing — it's drift. And that risk is now mitigated by the shared functions being the single source of truth. If someone adds a new `ServiceInfo` field, they update `serviceRecordToInfo` (one place) and both paths get it. If someone adds a new scope field, they update `scopeMeta` (one type) and both paths use it.

The architecture is done. The sharing is at the right level. Ship it.
