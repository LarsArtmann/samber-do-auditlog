06Open Questions

Q1: Should live/ be a separate Go module (like go-workflow-auditlog/live/) or stay in the same module? go-workflow-auditlog/live/ is a separate go.mod with replace directives. samber-do-auditlog explicitly does NOT modularize (AGENTS.md: "Do NOT modularize"). The live/ sub-package currently lives in the same module, which means it can access unexported types. If separate, it could only use exported APIs. Which approach?
Q2: Should the demo use the existing example/ ride-sharing domain or create a simpler one? The example/ has 19 features with a ride-sharing domain. A simpler demo (3-4 services) might be clearer for showing the live dashboard. Or extend the existing example to also serve the live dashboard?
Q3: Should the live dashboard reuse the templ-based HTML or build its own? The static dashboard (html.templ) is 1000+ lines with all CSS/JS inline. The live dashboard has its own separate template, CSS, and JS. Sharing would reduce drift but increase complexity. go-workflow-auditlog keeps them separate.
